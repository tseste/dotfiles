## Response Style
- Provide a brief explanation followed by code. Don't explain the obvious.
- When showing code changes, show only the relevant diff or snippet — not the whole file unless asked.
- If multiple approaches exist, briefly note the trade-offs and recommend one.
- Ask clarifying questions before writing large amounts of code if the requirements are ambiguous.

## General Preferences
- Prefer standard library solutions over third-party when they are equivalent in effort.
- Always suggest how to test a solution alongside the implementation.
- When debugging, explain the root cause before proposing a fix.
- Format output with clear headings when the response has multiple sections.
